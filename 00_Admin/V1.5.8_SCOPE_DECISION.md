# V1.5.8 Scope Decision

## Release Name

```text
v1.5.8 — AI Repo Execution, Recovery and Release Safety Governance Patch
```

## Decision

v1.5.8 is an additive governance patch to v1.5.7.

It does not replace the Short Video Lead Kit Partner Governance added in v1.5.7.

It adds the daily AI development execution safety layer needed for:

1. repo/workspace-based AI implementation
2. full repo context with issue-scoped modification
3. interruption recovery and WIP preservation
4. failed issue recovery and rollback
5. merge readiness and testing level validation
6. issue size and splitting control
7. environment secrets and API key safety
8. mock/demo/client data governance
9. preview/staging/production deployment gates
10. database migration and RLS change control

---

## Why This Release Is Needed

Optimaks is now using multiple AI tools across planning, coding, UI prototyping, research and review:

- ChatGPT
- Codex
- Antigravity
- Gemini
- GitHub
- Vercel
- Supabase

The previous governance already covered:

- GitHub Issue / Branch / PR / Preview governance
- Gemini research intake
- strategy review
- strategic guardrails
- one-person team operating model
- short video and partner governance

However, daily AI implementation exposed three practical risks.

---

## Risk 1 — ZIP-Based Development Causes Version Drift

If MVP01 or the development constitution is repeatedly passed around as ZIP files, AI tools may operate on a stale snapshot instead of the current GitHub repo.

This can cause:

- duplicated components
- mismatched issue dependencies
- outdated constitution rules
- accidental rewrite of existing structure
- inconsistent AI prompts
- harder debugging

Decision:

```text
Daily AI implementation should use GitHub repo or local workspace, not repeated ZIP uploads.
```

ZIP files remain valid only for:

- archive
- backup
- handover
- batch review
- fallback when repo access is unavailable

---

## Risk 2 — Interrupted AI Work Can Be Lost

AI-assisted implementation may be interrupted by:

- founder leaving the workstation
- laptop battery running out
- Codex usage limit reached
- Antigravity usage limit reached
- AI session instability
- internet instability
- switching tools mid-issue

Decision:

```text
Any interrupted AI implementation work must be preserved through branch, WIP commit, push when possible, and issue progress log.
```

---

## Risk 3 — Working Code Is Not Automatically Safe to Merge

AI-generated work may appear to run while still being unsafe due to:

- hidden scope creep
- modified forbidden files
- secrets exposure
- weak testing
- mock/demo/client data confusion
- database/RLS impact
- deployment gate bypass
- difficult rollback path

Decision:

```text
AI-generated work must pass merge readiness, testing level, secrets safety, data safety, deployment gate and founder review before merge or delivery.
```

---

## v1.5.8 Core Rules

```text
GitHub repo = central source of truth.
Local workspace = active construction site.
Full repo context may be read.
Only the assigned issue scope may be modified.
Clone once; pull updates.
Use one branch per issue.
Save interrupted work with WIP commit and progress log.
Resume from current branch; do not restart from scratch.
Failed AI work must be recoverable.
Merge requires a readiness gate.
Secrets must never enter prompts, frontend code, Git history, screenshots or public demos.
Testing level must match change risk.
Large issues must be split before execution.
Demo data must be clearly separated from real client data.
Production deployment requires explicit founder approval.
Database and RLS changes require isolated, reviewable issues.
```
